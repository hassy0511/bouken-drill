## 🔴 P0：Service Worker のキャッシュ更新問題【最優先・開発者が明確に不満を表明】

### 問題
コードを更新して GitHub Pages にデプロイしても、iPad 側で古いバージョンが表示され続けることがある。
開発者は毎回 **Safari の「Webサイトデータを削除」を手動実行**する羽目になっており、
「アップデートするたびにこの処理するのめっちゃ面倒」と明言している。

### 現状の実装（`sw.js`）
- `install` で `skipWaiting()`、`activate` で `clients.claim()` は呼んでいる
- `fetch` は **cache-first**（キャッシュがあればネットワークを見に行かない）
- `CACHE` 定数を手動インクリメントすることで古いキャッシュを消す運用

### なぜ問題が起きるか
- cache-first のため、`index.html` 自体が古いキャッシュから返され続ける
- Service Worker 本体の更新チェックはブラウザ任せで、確実に走る保証がない
- 開いている最中に新SWが有効化されても、ユーザーが見ている画面は古いまま

### あるべき実装（提案。設計から相談すること）
1. **`index.html`（ナビゲーションリクエスト）だけ network-first** にする
   （オフライン時はキャッシュにフォールバック）。SVG/アイコン等の静的資産は cache-first のままでよい
2. **更新検知 → 子ども向けの更新UI**：
   `registration.addEventListener('updatefound')` で新バージョンを検知し、
   「✨ あたらしい ぼうけんが とどいたよ！」というトースト＋「よみこむ」ボタンを表示。
   タップで `skipWaiting()` → `location.reload()`
3. **起動時に必ず更新チェック**：`registration.update()` を明示的に呼ぶ
4. 目標：**開発者が手動でキャッシュ削除する作業をゼロにする**

### 検証方法
- 旧バージョンをキャッシュさせた状態で新バージョンをデプロイし、
  リロードだけで新版に切り替わることを確認する
- オフライン時に従来通り起動できることを確認する（デグレ防止）

## 🟠 P2：ステージ・出題タイプの追加

子どもの学習が早いため、コンテンツが枯渇しやすい。実際 v4 の時点で
「最後のとうまで行ったが次のステージがない」と言われて 6 ステージ追加した経緯がある。

### 未実装の学習範囲（要件定義より）
- **算数**：あまりのあるわり算、かけ算の筆算、長さ・かさ・重さの単位、
  分数・小数の入口、図形（三角形・四角形・円）
- **国語**：短文読解（だれが・なにを・どうした）、主語述語、
  2〜3年の漢字、数え方（助数詞）

### 新パターンを追加する場合
文章題の `PATTERNS` に `mul` / `div` / `diff`（ちがいはいくつ）などを足すと、
`words.json` 側のテンプレート追加だけで問題を増やせるようになる。**費用対効果が高い。**

### ステージ追加の手順（チェックリスト）
1. シーンSVG（`viewBox 0 0 120 120`、`class="bd-arch"` 必須）を描き `SCENE_SVGS` に登録
2. 宝物SVG（`viewBox 0 0 100 80`、`class="scroll-wrap"`）を描き `TREASURE_SVGS` に登録
3. `STAGES[]` に追加（id/name/scene/theme/intro/subFlavor/gen/n/treasure）
4. `AREAS[]` を更新（新エリアなら追加、既存エリアの拡張なら `to` を更新）
5. `XS[]`（ジグザグX座標）が新ステージ数をカバーしているか確認
6. テーマCSS（`--card-bg`, `--card-frame`）＋コーナー装飾SVGを追加
7. コントラスト実測（AA 4.5:1 以上）
8. jsdom で通し検証
9. `sw.js` のキャッシュ番号をインクリメント

## 🟡 P3：漢字の手書き書き取り【当初要件・未着手】

要件定義の段階で開発者が「**書き取りテスト中心（本格）**」を選択した機能。
実装コストが高いため Phase 2 として保留され、現在も未着手。

### 設計方針（合意済み）
- 「今どの字を書かせているか」は既知なので、**汎用の手書き認識は不要**。
  お手本ストロークとの **1対1照合** で済む（オフライン完結可能）
- 入力：Canvas 2D ＋ Pointer Events（指／Apple Pencil）
- お手本データ：**KanjiVG**（オープンな書き順SVGデータ、CC BY-SA）の活用を想定
- 判定：画数 → 書き順（開始位置・方向）→ 各画の形状類似度（DTW等）→ 全体バランス
- **初期スコープでは トメ・ハネ・ハライの厳密判定はしない**（コストが跳ね上がるため）
- 「なぞり書き（書き順ガイド）」モードも用意し、練習→テストの導線にする

### 注意
KanjiVG を使うとライセンス表記（CC BY-SA）が必要になる。
また、データサイズがオフラインキャッシュに与える影響を検討すること。

## 🔵 P5：技術的負債・改善余地

- **`index.html` が約1,640行**と大きい。ただし**単一ファイル構成は意図的な設計判断**
  （ビルドレス運用のため）。分割を提案する場合は、
  GitHub にドラッグ&ドロップでデプロイする運用を壊さないことが絶対条件
- **確認用HTML**（全ステージ解放版）を毎回手作業で生成している。
  URLパラメータ（`?unlock=all`）で切り替える方式にすれば運用が楽になるかもしれない
  （ただし子どもが偶然発見しないよう配慮が要る）
- **テストが書き捨て**。jsdom ベースの検証スクリプトをリポジトリに常設し、
  回帰テストとして残す価値はある（ただしビルドレス方針との整合を考えること）

## 開発フロー（このプロジェクトの進め方）

1. 開発者が課題・要望を提示（子どもの実使用からのフィードバックが多い）
2. **設計を先に固める**（規模が大きい場合は設計書を作る。過去に4本作成している）
3. **開発者の「GO」を待つ**
4. 実装 → jsdom検証・コントラスト実測・SVGのPNG目視
5. 「確認用（全ステージ解放）HTML」を提供 → 開発者が手元のPCで確認
6. GitHub Pages にデプロイ（開発者がブラウザ操作で6ファイルをアップロード）
7. iPad で子どもが実使用 → フィードバック → 1に戻る
