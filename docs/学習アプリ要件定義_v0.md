# 子ども向け学習アプリ 要件定義（v0）

> 実装に入る前の合意用ドキュメント。★印は「こちらで仮決めした／要確認」の項目。

---

## 1. 目的・コンセプト

- 子ども（現在1年生・学習が早い）が **楽しみながら自走して** 算数・国語を学べる自作アプリ。
- 学年を先取りできるよう **1〜3年生** をスコープに含める。
- 学習スタイルは **ドリル × ゲーム世界観のバランス型**。反復練習を「冒険・探検」の文脈で包んで没入させる。

## 2. 対象・利用環境

- 利用者：子ども本人（**1人で自走**する前提。将来きょうだい対応の余地は残す ★）。
- 端末：**iPad**（タッチ／Apple Pencil）。
- ネット：**オフラインでも動く**こと（外出先・電車を想定）。
- 保護者（＝開発者）が進捗を見られる **親用ダッシュボード** を用意。

## 3. デプロイ・技術方針

- **配信**：GitHub Pages（または Cloudflare Pages）。無料・HTTPS・サーバ運用ゼロ。
- **PWA化**：`manifest.json` ＋ Service Worker。
  - ホーム画面に追加 → フルスクリーンでネイティブ風、ブラウザUIが消えて誤操作しにくい。
  - Service Worker でアセット・問題データをキャッシュ → **オフライン動作**。
- **時間制限**：iPadの「スクリーンタイム」「アクセスガイド（1アプリ固定）」と併用する運用を前提（アプリ側で強制はしない）。
- **技術スタック ★**：ビルドレス志向で **バニラ HTML/CSS/JS**（フレームワークなし）を想定。
  - Reactにしたい場合は要相談（ビルド環境が必要になる）。
- **データ保存**：軽量な進捗は `localStorage`、手書きストローク等の大きめデータは `IndexedDB`。
  - ※Claudeのアーティファクトで試作する時だけ localStorage 不可 → その場合はstate代替。実デプロイ版は通常どおり使用。

## 4. 全体構成（画面・モード）

冒険・探検（RUIN BUSTER的）の世界観でラップする。既存ゲームの `STAGES` 配列アーキテクチャと同発想。

```
[ホーム / 冒険マップ]
   └─ エリア（科目×学年）… 例：算数エリア／国語エリア
        └─ ステージ（単元ごとの問題セット）
             └─ 問題（出題）→ 即時フィードバック
        └─ ボスステージ（単元まとめテスト）
[図鑑 / たからばこ]  … 収集した「おたから」「かせき」等
[きろく]            … 星・クリア状況
[保護者メニュー]     … PIN保護つきダッシュボード ★
```

- **進行を詰まらせない**：正解が続くと上の学年ステージが解放（先取り）。間違えても優しく再挑戦。
- 星3段階評価でやり込み要素。

## 5. 学習コンテンツ範囲

### 算数
- **1年**：数（〜20/合成分解・大小）、たし算・ひき算（繰り上がり／繰り下がり）、とけい（何時・何時半）
- **2年**：たし算ひき算の筆算、**九九**、100より大きい数（〜1000）、長さ(cm/mm)・かさ(L/dL/mL)、時刻と時間、三角形と四角形、分数の素地
- **3年**：わり算・あまりのあるわり算、大きな数（万）、3桁の筆算・かけ算の筆算、長さ(km)・重さ(g/kg)、円と球、小数・分数の入口

### 国語
- **漢字**：1年80字／2年160字／3年200字（**読み**と**書き**）
- **ことば**：反対語、仲間のことば、数え方、カタカナ語、主語・述語
- **読解**：短文の読み取り（だれが・なにを・どうした）

## 6. 出題形式と自動採点

| 形式 | 用途 | 採点 |
|---|---|---|
| 4択選択 | 漢字の読み、ことば、読解 | 即時 ○× |
| 数値入力（テンキー） | 計算、九九、単位 | 即時 ○× |
| マッチング／線つなぎ | 反対語、単位、仲間集め | 即時 ○× |
| 並べ替え | 文の順序、数の大小 | 即時 ○× |
| タイムアタック | 九九・計算フラッシュ | 正答数＋タイム |
| **手書き** | 漢字の書き取り／なぞり | ストローク照合（§7） |

## 7. 漢字の手書き（書き取り）仕様 ★重点

方針：**書き取りテスト中心（本格）**。「今どの字か」は既知なので、汎用手書き認識ではなく**お手本テンプレートとの1対1ストローク照合**で実装する（オフライン完結）。

- **入力**：Canvas 2D ＋ Pointer Events で指／Apple Pencilのストローク（点列）を取得。
- **お手本データ ★**：各字の「画ごとのストローク（点列・書き順）」が必要。手作業定義は非現実的なため、**KanjiVG 等のオープンな書き順ストロークデータ（SVG）の活用を想定**（ライセンス：CC BY-SA、要確認）。
- **判定要素**：
  1. 画数が合っているか
  2. 各画の書き順（開始位置・方向）
  3. 各画の形状の類似度（正規化してDTW／点列距離で照合）
  4. 全体の配置バランス
- **採点レベル（初期）★**：**書き順＋おおまかな字形の一致**まで。**トメ・ハネ・ハライの厳密判定は初期スコープ外**（実装コストが跳ね上がるため）。○／△／×でフィードバック。
- 補助として「なぞり書き（書き順ガイド）」モードも用意し、練習→テストの導線にする。

## 8. ゲーム・報酬システム

- **世界観**：遺跡・島の探検（RUIN BUSTER的なやさしい版）。問題を解くと先へ進める。
- **報酬**：クリアで「おたから」「かせき」「どうぐ」等を収集 → **図鑑／たからばこ**が埋まる。
- **評価**：ステージごとに星1〜3。
- **演出**：**音声は控えめ**（BGMは控えめ／オフ可、効果音は最小限）。キャラのリアクションやアニメなど**視覚演出中心**。

## 9. UI/UX 方針

- **一人で自走できる導線**：全文 **ふりがな** つき（読めない字で詰まらせない）。説明は文字＋アイコン。
- **音声控えめ**：読み上げは補助的なオプション扱い（デフォルト控えめ／オフ）。
- 大きめのタップ領域、シンプルなナビ、「もどる／やめる」が分かりやすい。
- フルスクリーンPWAで、誤操作でアプリ外に出ない設計。

## 10. データ設計・進捗管理

- **問題データ**：`STAGES` 的なデータ駆動（JSON）。
  - **オーサリング ★**：問題は **Excelでマスタ管理 → JSONへ変換して同梱**（既存のSheetJS運用と同じパターン。コードを触らず問題を追加可能）。
- **進捗**：単元ごとの正答率・実施回数・最終学習日・つまずき記録。
- **プロフィール**：初期は1人。データ構造上は複数プロフィールに拡張しやすくしておく ★。
- **親ダッシュボード**：苦手単元、学習時間、進み具合を可視化。**PIN保護**で子どもが入れないように ★。

## 11. 初期問題セットのボリューム（★提案）

初手からそこそこの量を同梱する。目安：

- 算数：各単元 **20〜40問**（計算・九九は多め）。
- 国語（漢字）：**学年配当を網羅**（読み＋書きの両モード）。
- 国語（ことば・読解）：単元ごとに **15〜30問**。

## 12. 実装フェーズ計画（★提案）

- **Phase 0（今ここ）**：要件確定 → PWAの殻・ナビ・ダミー問題1つで骨組み確認。
- **Phase 1（MVP）**：算数の計算＆九九／漢字の読み＋なぞり／選択・数値入力／進捗保存／冒険マップ最小版。
- **Phase 2**：漢字の本格書き取り（KanjiVG照合）／読解／報酬・図鑑／親ダッシュボード。
- **Phase 3**：2〜3年の問題セット拡充／タイムアタック／演出強化。

## 13. こちらで仮決めした点（要確認まとめ）

1. 技術スタックは **バニラJS＋PWA**（フレームワークなし）。Reactにしたいか？
2. 手書きお手本は **KanjiVG等のオープンデータ**を活用（ライセンスCC BY-SA前提）。
3. 書き取り採点は **書き順＋字形一致まで**（トメハネハライは対象外）。
4. プロフィールは **初期1人**（複数は将来拡張）。
5. 問題の初期ボリューム目安（§11）とフェーズ計画（§12）。
6. 問題データは **Excelマスタ → JSON同梱** の運用でよいか。
