## 1. プロジェクト概要

**何**：小学1年生の子ども（学習が早く、2〜3年生の内容も先取り可能）のための、
算数・国語の学習ゲーム。「島を探検する」冒険の文脈で、ドリル学習をゲーム化している。

**誰が使う**：開発者本人の子ども（1人）。iPad で毎日プレイ中。実際にテスト運用しながら育てている。

**現状**：v10。v9で「島＝1カリキュラム」制へ移行し、現在は「たしざんの しま」11ステージが稼働中。v10ではホーム、マップ、クイズ、結果、図鑑、保護者画面を全面刷新し、バリア・コンボ・きらめき・次ステージ導線を実装した。GitHub Pagesで公開し、iPadにホーム画面追加（PWA）して使用する。最新一覧は`docs/stage_overview.md`、デザイン判断は`REDESIGN_AUDIT.md`が正。

**デザインの原則（重要）**：
初期バージョンは絵文字アイコン＋ベタ塗りで「しょぼい」と却下された経緯がある。
現在は **輪郭線つき・2階調のセルシェーディング風SVGイラスト** に全面刷新済み。
**この品質基準を下げないこと。** 新しいステージや要素を足す際も同じ水準で描くこと。

v10以降の体験原則は **「上質な絵本 × 海賊の冒険ボードゲーム」**。問題を解く行為が、敵のバリア、コンボ、宝物、次の行き先へ画面上でつながること。単に機能やカードを追加するのではなく、「目的→挑戦→反応→報酬→次」のゲームループを保つこと。

## 2. 技術スタック・制約（厳守）

| 項目 | 内容 |
|---|---|
| 構成 | **単一HTMLファイル**（`index.html`）にHTML/CSS/JSすべて内包 ＋ `words.json`（問題データ） |
| フレームワーク | **なし**（バニラJS）。React等の導入禁止。ビルド工程を持たない |
| 外部ライブラリ | **禁止**。CDN含め一切追加しない |
| 画像 | **すべてインラインSVG**（JS内のテンプレート文字列）。ラスター画像・AI生成画像の追加禁止 |
| アニメーション | **CSSのみ**（`@keyframes` + `transform`/`opacity`）。JSアニメライブラリ禁止 |
| ホスティング | GitHub Pages（`https://hassy0511.github.io/bouken-drill/`） |
| 配信形態 | PWA（`manifest.json` + `sw.js`）。オフライン動作必須 |
| 保存 | `localStorage`（進捗）。バックアップはJSONエクスポート/インポート |
| 対応環境 | iPad Safari が本番。PCブラウザでの `file://` 直開き確認もサポートすること |

### 制約の理由
- 単一ファイル＋ビルドなし＝GitHubにドラッグ&ドロップでデプロイできる運用にしているため
- 外部ライブラリなし＝オフライン動作とファイルサイズのため
- これらは意図的な設計判断。**「モダン化」や「リファクタで分割」を勝手に提案しないこと**

## 4. アーキテクチャ：2層分離（重要な設計思想）

問題の「ロジック」と「素材データ」を意図的に分離している。**この分離を壊さないこと。**

### 層1：出題ロジック（`index.html` 内のJS）
- 「何を、どう問うか」の型。`gen*()` 関数群がこれにあたる
- 例：`genAdd10()`（たし算）、`genClock()`（時計よみ）、`makeKanjiGen(poolName, prompt)`（漢字4択）
- **新しい出題方式を増やす** ＝ ここに新しい `gen` 関数を追加する

### 層2：素材データ（`words.json`）
- 「型」に流し込む中身。語彙リスト、文章題テンプレート
- **同じ出題方式のまま量を増やす** ＝ ここに配列要素を足すだけ（コード変更不要）
- 参照は必ず遅延評価で行う（`poolOf('g1a')` のように名前で引く）。
  理由：`words.json` は非同期ロードのため、起動直後は未ロードの可能性がある

### ロード戦略（重要な落とし穴）
- `words.json` は `fetch()` で読む。**しかし `file://`（PCでダブルクリック）では CORS で必ず失敗する**
- そのため `index.html` 内に `WORDS_FALLBACK`（最小限の内蔵データ）を持ち、失敗時はそれで動く
- **新しい語彙カテゴリを追加したら、フォールバックにも最小限の同カテゴリを追加すること**
  （そうしないと file:// 確認時にそのステージが動かない）
- `startQuiz()` は `wordsReady` Promise を待ってから出題を構築する

## 5. 主要なデータ構造

### `STAGES[]`（`index.html` 内）— ステージ定義。データ駆動の中核

```js
{
  id: 'mori',                    // 一意ID。進捗保存のキーになる（変更禁止：既存ユーザーの進捗が飛ぶ）
  name: 'すうじの もり',          // マップ表示名
  icon: '🌳',                    // フォールバック用（scene未登録時のみ使用）
  color: 'var(--ocean)',         // ノード枠色
  scene: 'mori',                 // SCENE_SVGS のキー。マップノードのイラスト
  theme: 'mori',                 // data-stage-theme 属性値。クイズ画面の配色
  intro: '…',                    // ステージ開始時の導入モーダル文言（同日1回だけ表示）
  subFlavor: '…',                // 問題文の下のヒント文（省略可）
  clearTitle: '…',               // クリア時の特別タイトル（ボスのみ。省略時は「〇〇 クリア！」）
  boss: true,                    // ボスフラグ。クリア時に花火演出が出る
  gen: genCompare,               // 出題関数（層1）
  n: 10,                         // 出題数
  treasure: {
    i: '🌰',                     // フォールバック絵文字
    n: 'もりの どんぐり',          // 宝物名
    scene: 'acorn',              // TREASURE_SVGS のキー
    memo: '…'                    // ずかんに表示される豆知識
  }
}
```

### `AREAS[]` — マップのエリア帯（ゾーン）
ステージのインデックス範囲で区切る。マップ上の色帯とラベルが**動的生成**される。
ステージを追加したら **必ず AREAS も更新すること**（しないと新ステージがどのエリアにも属さない）。

### `SCENE_SVGS` / `TREASURE_SVGS` — SVGアセットのレジストリ
`scene` 名 → SVG文字列 のマップ。新ステージ追加時はここに登録する。

### `PATTERNS` — 文章題の計算パターン（層1）
add / sub / mul / div / diff を実装済み。div は「答えが整数」ガードで割り切れを保証
（違反時は再生成）。**さらに増やす時はここにエントリを足す。**
`words.json` 側が未知のパターンを指定していた場合は、そのテンプレートをスキップして
`console.warn` する（エラーで止めない）設計。

## 6. SVGアセットの作法（品質基準）

新しいシーン・宝物を描く際は、**必ず既存アセットと同じ作法**に従うこと。

- **すべての図形に輪郭線**：`stroke="#3D3323"`（Ink Line）、`stroke-width="1.5〜2.5"`、`stroke-linejoin="round"`
- **単色ベタ塗り禁止**：1つの立体につき最低2階調（明部・暗部）を別パスで塗り分ける（セルシェーディング）
- **グラデーション不使用**
- **石や岩は完全な長方形にしない**（四隅を数px歪ませる）
- **光源は右上45度で統一**
- シーンSVG：`viewBox="0 0 120 120"`
- 宝物SVG：`viewBox="0 0 100 80"`、ルート要素に `class="scroll-wrap"`（pop演出用）
- **シーンSVGは必ず光る要素を1つ持ち、`class="bd-arch"` を付ける**
  （挑戦中のglow／クリア時の金色化のCSSがこのクラスを対象にしている。付け忘れると演出が出ない）
- 宝物で「開く」等の2状態演出をしたい場合：`class="tr-a"`（初期）と `class="tr-b"`（変化後、
  `style="display:none;opacity:0"`）の2グループを持たせる。汎用機構が自動でクロスフェードする

### ナビキャラ「コンパス」
- `BUDDY_SVG` の1つのSVG内に、表情4種（`bd-face-normal/happy/oops/super`）を `<g>` で持つ
- 切替は `setBuddyFace(el, 'happy')` を呼ぶだけ。針の回転は `transform-origin: 50px 50px` 厳守
- 名前は子どもが初回起動時に付ける（`state.buddyName`）。セリフは `withName()` でラップする

## 7. アニメーション仕様（タイムライン厳守）

フィードバックの表示時間とCSSアニメの尺は**噛み合わせて設計されている**。片方だけ変えないこと。

| 場面 | 総時間 | 内訳 |
|---|---|---|
| 正解 | `flashFB(1020)` | 0ms:フェードイン(120ms)＋ボタンバウンド(300ms)／0ms:針1回転(500ms)／100ms:パーティクル(600ms)／900ms:フェードアウト |
| 不正解 | `flashFB(1920)` | 0ms:針の首振り(400ms)＋カードシェイク(300ms)／300ms:正解テキストのフェードイン／1800ms:フェードアウト |
| 宝物発見 | — | 0ms:巻物pop(500ms)／600ms:tr-a→tr-bクロスフェード(400ms)／1000ms:ナビキャラsuper登場 |

`prefers-reduced-motion: reduce` で全アニメを無効化する記述が末尾にある。
**新しいアニメを足したら、必ずこのメディアクエリにも追加すること。**

## 8. デプロイと Service Worker（既知の課題あり）

### 手順
GitHub のリポジトリ画面 → Add file → Upload files → 6ファイルをドラッグ → Commit changes。
（開発者は git コマンドを使わず、ブラウザ操作でデプロイしている）

### キャッシュ更新のルール
**コードを変更したら `sw.js` の `CACHE` 定数を必ずインクリメントすること**（`bouken-v5` → `v6`）。
これを忘れると iPad に変更が反映されない。

### 【解決済み（v6〜）】
network-first(HTML/words.json) + 更新トースト「✨あたらしい ぼうけんが とどいたよ！」を実装済み。
新バージョンはトーストのタップだけで反映される。手動のキャッシュ削除は不要。
**コード変更時の `CACHE` インクリメントは引き続き必須。**

## 9. 検証（実装後に必ず実施）

ブラウザが無い環境で開発しているため、以下で品質を担保している。**同じ手順を踏襲すること。**

1. **jsdom による通し検証**：全ステージについて「導入モーダル→出題→テーマ属性→正解/不正解演出→
   クリア→宝物演出→ずかん反映」が例外なく動くか
2. **出題ロジックの値域テスト**：各 `gen` 関数を1000〜3000回実行し、
   答えがマイナスにならない／範囲内／選択肢が重複しない／正解インデックスが正しい ことを検証
   （特に文章題は「答えが0未満・1000以上にならない」ことを必ず確認）
3. **コントラスト実測**：文字色×カード背景が WCAG AA（4.5:1）以上か、相対輝度計算で検証
4. **SVGの目視確認**：`cairosvg` でPNG化してコンタクトシートを作り、破綻がないか目で見る
5. **後方互換**：既存ステージの見た目・挙動が変わっていないこと。既存の進捗データが読めること

## 10. やってはいけないこと

- ステージの `id` を変更する（既存ユーザーの進捗が飛ぶ）
- `localStorage` のキー `bk_state` を変更する（同上）
- 外部ライブラリ・ビルドツールの導入
- 絵文字をアイコン代わりに使う（品質基準の後退。フォールバック用途を除く）
- 単色ベタ塗りのSVG（輪郭線と2階調は必須）
- 子どもを否定する文言（不正解時も励ます。「まちがい」より「おしい！」）
- 難しい漢字を使ったUI文言（子どもが1人で読むので、ひらがな中心・分かち書き）
